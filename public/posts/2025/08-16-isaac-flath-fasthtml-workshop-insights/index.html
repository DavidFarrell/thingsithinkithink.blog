Key Insights from Isaac Flath's FastHTML Workshop: Building Annotation Tools That Scale

Deep dive into the practical lessons and technical patterns from Isaac Flath's workshop on building custom annotation tools with FastHTML and HTMX.

  • David Gérouville-Farrell
  • August 16, 2025
  • 4 min read

This is my second post today covering Isaac Flath’s excellent FastHTML workshop from the AI Evals course. While my first post covered the high-level concepts and use case, this one dives deeper into the technical implementation details and workshop insights that I found particularly valuable.

Workshop Structure and Real-World Context

Isaac’s workshop was structured around a real-world use case: building annotation tools for AnkiHub, a medical flashcard platform. This wasn’t a theoretical exercise - it was solving actual problems they faced in production.

The key insight here is that domain-specific annotation tools always beat generic ones. When you’re dealing with medical terminology, technical precision matters more than similarity. “Purine” and “pyrimidine” might be semantically related, but they’re completely different concepts that require different flashcards.

FastHTML Philosophy: Context Switching is Expensive

Isaac emphasized that FastHTML’s main benefit isn’t just simplicity - it’s about staying in your mental context. If you’re already thinking in Python, why switch to HTML/CSS/JavaScript?

Here’s the minimal example he showed:

from fasthtml.common import *
from monsterui.all import *

app, rt = fast_app(hdrs=Theme.blue.headers())

@rt
def index():
    return Container(H1("Hello World"))

serve()

The @rt decorator creates routes based on function names. index() becomes your / route. No separate files, no build steps, no context switching.

Database Design with FastLite

The annotation system uses a simple but effective schema:

from fastlite import Database

db = Database('eval.db')

@dataclass
class Annotation:
    input_id: str
    document_id: int
    input: str = None
    document: str = None
    eval_type: str = None  # 'good', 'bad', or None
    notes: str = None
    reviewer: str = None

FastLite provides the same minimal-code philosophy for databases that FastHTML provides for web apps. You get SQLite’s simplicity with Python-native access patterns.

The Main Dashboard: Aggregating Distinct Inputs

Isaac’s main page shows a clever pattern for handling the “inverted retrieval” problem:

@rt
def main():
    # Get all unique input queries
    distinct_inputs = db.annotations.select_distinct('input_id', 'input')
    
    # Create action buttons for each input
    actions = []
    for input_id, input_text in distinct_inputs:
        actions.append({
            'input': input_text,
            'action': Button("Evaluate", 
                           hx_get=evaluate.to(input_id=input_id),
                           cls=ButtonT.primary)
        })
    
    return Container(
        H1("Annotation Dashboard"),
        V.table_from_dict(actions, ['Input', 'Action'])
    )

The key insight here is using select_distinct to avoid showing duplicate queries. Each unique input gets one row with an evaluate button.

HTMX: The Magic of Partial Updates

Isaac demonstrated several HTMX patterns that make the interface feel modern:

Auto-Save Notes

def notes_field(input_id: str, document_id: int, current_notes: str = ""):
    return TextArea(
        current_notes,
        placeholder="Add evaluation notes...",
        hx_post=update_notes.to(input_id=input_id, document_id=document_id),
        hx_trigger="change",
        cls="min-w-96"
    )

The hx_trigger="change" means notes save automatically when users finish typing and tab away. No submit button needed.

Dynamic Button Updates

@rt
def evaluate_doc(input_id: str, document_id: int, eval_type: str):
    # Update database
    db.annotations.update(Annotation(
        input_id=input_id, 
        document_id=document_id, 
        eval_type=eval_type
    ))
    
    # Return updated button HTML
    return eval_buttons(input_id, document_id)

When users click Good/Bad, HTMX replaces just the button container with updated styling. The clicked button becomes primary (blue), the other becomes secondary (gray).

Button State Management Pattern

The evaluation buttons show a sophisticated state management pattern:

def eval_buttons(input_id: str, document_id: int = None):
    target_id = f"#eval-{input_id}-{document_id}"
    _annotation = db.annotations[input_id, document_id]
    
    def create_eval_button(label: str):
        is_selected = _annotation.eval_type == label.lower()
        return Button(
            label,
            hx_post=evaluate_doc.to(
                input_id=input_id, 
                document_id=document_id, 
                eval_type=label.lower()
            ),
            hx_target=target_id,
            cls=ButtonT.primary if is_selected else ButtonT.secondary,
            submit=False
        )
    
    return DivLAligned(
        create_eval_button("Good"), 
        create_eval_button("Bad"),
        id=target_id[1:]
    )

Each button checks if it matches the current database state. If yes, it shows primary styling. If no, secondary styling. This creates immediate visual feedback without complex JavaScript state management.

Keyboard Shortcuts: When They Matter Most

Isaac’s keyboard shortcuts are most valuable when individual tasks are quick. If you’re reviewing 400 flashcards at 2 seconds each, removing mouse navigation saves significant time. If each review takes 5 minutes, keyboard shortcuts matter less.

The lesson: Optimize for your actual usage patterns, not theoretical best practices.

Deployment and Hosting Options

Isaac mentioned several deployment options:

  1. Plash: FastHTML’s new hosting service
  2. Any Starlette-compatible hosting: Since FastHTML builds on Starlette
  3. Modal: For serverless deployment
  4. Traditional VPS: For full control

The key point is that FastHTML doesn’t lock you into a specific hosting solution.

Key Takeaways

  1. Domain-specific tools always win over generic annotation platforms
  2. Context switching is expensive - FastHTML keeps you in Python
  3. HTMX enables modern UX without JavaScript complexity
  4. Simple database schemas with FastLite scale surprisingly well
  5. Keyboard shortcuts matter most for high-volume, quick tasks

The workshop reinforced that building custom tools isn’t just about technical implementation - it’s about encoding your domain expertise directly into the interface. When you understand your specific use case deeply, you can build tools that generic solutions can’t match.
